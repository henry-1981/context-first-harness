# rules/workflow.md

모든 작업에 적용되는 행동 규칙. 메모리에 박제된 feedback 중 **자주 위반 + 도메인 무관 + 위반 시 손실** 기준으로 승격.

## 외부 소스 도입

### 클론 우선
- **원칙**: 외부 소스를 도입할 때는 **완전 클론이 기본**
- **"클론해와라" = `cp -r`** — 자체 재구현 금지, 원본 그대로 복사
- **이유**: 재구현은 미묘한 차이를 만든다. 레퍼런스의 가치는 원본 그대로일 때 최대

### 레퍼런스 사양 투영 금지
- **원칙**: 레퍼런스 제품·도구의 사양을 클라이언트 제품에 투영 금지
- **예**: "A 회사 SaMD는 Class II이니 이 회사도 Class II"는 틀린 추론
- **이유**: 같은 카테고리여도 세부 사양·위험도·적응증이 다르면 등급이 다르다

## 분석·관찰

### 관찰 우선, 결론 나중
- **원칙**: 과거 흔적·자료를 볼 때 **바로 전략·결론 재구성 금지**. 관찰부터 쌓기
- **순서**: 수집 → 관찰 → 패턴 식별 → 가설 → 결론
- **위반 시**: 초반 인상에 편향된 잘못된 전략

### 메타 결정 위임 금지
- **원칙**: 탐색 항목을 나열했으면 탐색 여부 자체를 사용자에게 메타 결정 요청 금지
- **해석**: "이것들을 탐색할까요?"가 아니라, 탐색이 필요하면 바로 하거나, 불필요하면 나열 자체 안 함

### 샘플 오버핏 금지
- **원칙**: 소수 샘플 fleet 안에 수치 끼워맞추기 금지
- **대안**: 일반론·방법론 차원으로 먼저 올라간 뒤 샘플 검증
- **위반 시**: 샘플에 맞춘 규칙이 일반 케이스에서 무너진다

## 의사결정 모드 구분

### 의견 vs 확정
- **의견 검토 모드**: 탐색적 질문, "어떻게 생각해?", closure 강요 금지
- **확정 결정 모드**: "이대로 하자", 실행 착수 승인
- **원칙**: 두 모드를 구분해서 응답. 의견 요청에 마치 확정된 것처럼 실행 금지

## 사용자-facing 응답 작성

### 압축 금지 — 사용자가 한 줄만 보고 yes/no 가능한 수준까지 풀어쓰기
- **원칙**: 옵션·결정·인사이트 제시 시 LLM 컨텍스트(spec·handoff 본문, 이전 세션 박제, bias-reviewer 권고, 메모리 등)를 사용자가 펼쳐들고 있다고 가정하지 않는다
- **이유**: LLM은 모든 문서를 동시에 들고 있으나 사용자는 그렇지 않다. 압축 = LLM 효율 / 사용자에겐 불해석 + 재질문 비용 + 스트레스. 풀어쓴 길이 비용보다 항상 작다
- **적용 규칙 4개**:
  1. **옵션 라벨 옆에 한 줄 평문 부연**. 예: "A: ready 그대로 두는 안", "B: blocked로 바꿔 외부 신호 대기 표시하는 안". 라벨만 던지지 않는다
  2. **본문 reference는 인용문 그대로**. "spec 결정 5번" → "spec의 'Gemini 모델 채택' 결정". "Case A" → "Case A (슬라이드 PDF 변환 fragment)"
  3. **인사이트는 프레임 용어가 아니라 구체 상황 서술**. "Static/Behavioral 갭" → "갖춘 자산은 있는데 실제 호출 흐름이 없다". "spec-페어 패턴" → "spec(영구 결정 박스) + handoff(작업 인계) 분리"
  4. **그리스 문자·내부 코드네임 라벨 단독 사용 금지**. (α)(β)(γ)·D1~D4·G1/G2/G3 같은 라벨은 옆에 무엇인지 부연. 라벨로 압축한 후 다음 응답에서 라벨만 인용하는 패턴 특히 위험
- **체크 질문**: 응답 작성 후 "사용자가 본문을 안 펼쳐도 이 한 줄만 보고 결정 가능한가?"를 자문. 아니면 풀어 다시 쓴다

### 연쇄 질문은 하나씩 — 앞 답이 뒷 질문의 옵션 집합을 바꾸면 분할 필수

- **원칙**: 파생 질문들이 서로 의존(앞 결정이 뒷 옵션을 좁힘) 관계면 **한 메시지당 한 질문**. 일괄 나열 금지
- **이유**: 일괄 나열 시 사용자는 각 질문을 독립인 양 답하게 되고, 뒤 질문 옵션이 앞 답으로 달라지는 경우가 마스킹됨. 결과적으로 잘못된 전제 위 결정이 누적된다
- **적용 규칙**:
  1. 파생 질문이 2개 이상이면 **의존성 그래프**(어느 답이 어느 질문의 전제를 바꾸는지) 먼저 짧게 제시
  2. 사용자에게 순서 승인 받고 **첫 질문만 펼침**
  3. 독립이라 확인된 질문만 묶어서 내거나, 병렬 처리 가능함을 명시
- **위반 감지 체크**: 응답에 번호 질문 3개 이상 + 답변이 다른 질문 옵션 집합에 영향 → 즉시 분할
- **"압축 금지"와의 관계**: 옵션은 풀어쓰되 질문 개수는 덜어낸다. 두 규율이 상호작용해 "풀어쓴 질문을 다수 나열"하는 결과가 되지 않도록 **먼저 분할, 다음 풀어쓰기** 순서
- **선행 사례**: 2026-04-18 documents→articles 파이프라인 브레인스토밍 세션, 파생 질문 6개 일괄 제시 → 사용자 교정. 본 규칙은 이 사례로 박제

### 허수 옵션 금지 — 실질 권장안이 명백하면 분기 자체를 제거

- **원칙**: 옵션들 간 결과가 사용자 선호에 따라 유의미하게 갈리지 않으면 옵션화하지 말고 **바로 권장 경로로 진행 + "이 경로로 갑니다" 통보**
- **이유**: "선택지 제시 = 친절"의 오해. 실제로는 사용자 결정 비용 증가 + 세션 공회전. 자명한 권장안을 "선택지 A"로 포장하면 사용자가 매번 "당연히 A지" 답하는 소모 반복
- **구분 체크 질문**: 옵션 펼치기 전 자문 — "사용자가 B나 C를 고를 **실제 이유**가 있는가?" 없으면 A 직행
- **lessons 2026-04-18 "스펙트럼 2-3지점 기본" 규칙과의 관계**: 그 규칙은 "극단 2지점보다 중간 지점을 더 탐색하라"는 의미이지 **"모든 결정에 3옵션을 강제"가 아님**. 권장안이 자명하면 옵션화 자체가 공회전
- **연쇄 질문 분할 규율과의 관계**: 연쇄 질문 분할은 "여러 독립 질문을 한 번에 던지지 말 것", 허수 옵션 금지는 "한 질문의 옵션을 가짜로 부풀리지 말 것". 둘 다 지키면 **진짜 선택지만 하나씩** 제시
- **선행 사례**: 2026-04-18 documents→articles plan 작성 후 "P-A(review 먼저) / P-B(사용자 육안) / P-C(바로 실행)" 3지점 펼침 → 사용자 교정 "그냥 P-A 했어야 하잖아". P-A가 자명한 권장 경로인데 허수 옵션으로 포장해 선택 강제. 본 규칙은 이 사례로 박제

## 문제 대응

### 즉시 수정 금지
- **원칙**: 사용자가 문제를 보고하면 **즉시 코드 수정 금지** (이미 ~/.claude/CLAUDE.md에 있음, 강조 목적)
- **순서**: 로그/데이터 수집 → 원인 분석 → 분석 결과 + 개선안 제시 → 사용자 확인 → 수정
- **금지**: 단편적 미봉책 반복. 근본 원인을 찾아 한 번에 해결

## 작업 종료

### 찌꺼기 남기지 않기
- **원칙**: 찌꺼기 남기지 말 것, 자료 관리가 품질에 직결
- **적용**: 임시 파일, 미완성 스크립트, 실험 브랜치, 테스트 덤프 모두 정리
- **트리거**: `/repo-cleanup` 스킬 주기 실행

## 법률·규정 분석

### 완독
- **원칙**: 법률 조항 분석 시 **면제·예외 규정까지 끝까지** 읽을 것
- **위반 시**: 예외에 해당하는 케이스를 일반 규정으로 잘못 적용

### 범위 엄수
- **원칙**: 계약서·규정 검토 범위는 **계약 문구에 한정**, 임의 확대 금지
- **예**: "업무 위탁" 계약에서 "품질관리"까지 언급하지 않음

## 스킬 수정

- **원칙**: 스킬 파일 수정 시 반드시 **skill-creator 거칠 것**
- **이유**: 스킬은 다른 세션·다른 사용자가 의존. 직접 편집은 검증 누락 + 시스템 파편화
- **예외**: typo 수정 등 명백한 단순 수정

## 산출물 검증

- 문서 산출물 완성 후 `deliverable-review` 스킬로 검증 게이트 통과
- 도메인에 맞는 reviewer 자동 디스패치 (meddev-compliance, spec, general)

## 보안

- 시크릿 파일 (`.env`, 서비스 계정 JSON 등) Read/cat/Bash로 **읽지 않는다**
- 환경변수 키 목록은 `process.env.` 패턴 grep
- 값이 필요하면 사용자에게 직접 입력 안내
- 실수로 출력됐으면 즉시 경고 + 키 교체 안내

### 평문 시크릿 박제 금지

- **원칙**: API 토큰·API 키·service account 값을 `CLAUDE.md`, `rules/`, `memory/`, `wiki/`, reference 문서 어디에도 **평문으로 박제하지 않는다**
- **이유**: `memory/*.md`, `CLAUDE.md` 계층, `rules/*.md`는 세션 시작 시 **자동으로 대화 컨텍스트에 주입**된다. 평문 박제 = 상시 노출
- **박제 규칙**:
  - 경로·파일명·환경변수명만 남긴다 (예: `ANTHROPIC_AUTH_TOKEN: settings.json env 참조`)
  - 실제 값은 `~/.claude/settings.json` env 섹션, OS keychain, `.env` 파일에만 둔다
  - 설명 문서에서도 placeholder(`<REDACTED>`, `<env 참조>`) 사용
- **위반 감지 시 조치**: 즉시 사용자에게 경고 → 토큰 재발급 안내 → 박제된 값 redaction → git log에 커밋돼 있으면 history rewrite 검토 (git filter-repo는 별도 backup/clone 필수 — `feedback_filter_repo_backup_strategy.md`)
- **선행 사례**: 2026-04-18 `memory/glm_claude_switch_guide.md`에 Z.AI 토큰 평문 박제 발견 → 사용자 재발급 + redaction 완료. 본 규칙은 이 사례로 박제
