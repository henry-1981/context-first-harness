# rules/content-writing.md

위키 페이지, SOP, 제안서, 발표자료, 보고서, LinkedIn 글 등 **문서 산출물** 생성 시 적용한다. 대화에는 적용하지 않는다.

## 번역투 금지

### 조사·문법
- ~에 의해 → 능동형
- ~에 대해 → 을/를
- ~에 있어서 → ~에서
- ~을 통해 → ~으로 / ~하면서
- ~로부터 → ~에서
- ~함으로써 → ~해서

### 동사
- 수행하다 → 하다
- 제공하다 → 주다
- 활용하다 → 쓰다
- 구축하다 → 만들다
- 가지고 있다 → 있다

### 구조
- 수동태 → 능동형
- ~의 ~의(소유격 중첩) → '의' 줄이기
- 불필요한 대명사(그/그녀/그들) → 생략 또는 구체 명사

### 종결어미
- "~것입니다" "~할 수 있습니다" 과다 금지
- "~다", "~된다", "~일 때가 많다" 등 섞어서 리듬 만들기

## 한영 병기

- 기술 용어 원어 병기는 첫 등장 1회만. 이후 한글만
- 한국 개발자 사이에서 통용되는 형태 우선 (deploy→배포, commit→커밋)
- 이미 한국어로 충분한 단어에 영문 괄호 붙이지 않기 (사용자(user) → 사용자)

## 금지 문체 (한국어)

### 구문 패턴
- "A가 아니라 B다" 구문 (앞 부정 + 뒤 재정의 패턴)
- 동일 문장 구조 3회 이상 반복
- 메타 설명 ("이 섹션에서는...", "살펴보겠습니다", "알아보겠습니다")

### 과장·공허 수식어
- 강력한, 획기적인, 완벽하게, 원활하게, 포괄적인
- 종합적인, 체계적인, 효율적인 (의미 없이 반복될 때)

### AI 도입부·결말 패턴
- "오늘날의 급변하는...", "디지털 시대에는..."
- "이 글에서는 X에 대해 알아보겠습니다"
- "~의 중요성은 아무리 강조해도 지나치지 않습니다"
- "지금까지 살펴본 바와 같이..." + 앞 내용 반복
- "이처럼 ~은 매우 중요한 의미를 가집니다"

### 전환어 남용
- "또한", "그러나", "따라서", "더불어", "한편" 문단마다 반복 금지
- 꼭 필요할 때만, 그래야 의미를 가짐

### 공허한 목록화
- 산문으로 쓸 수 있는 내용을 모두 글머리 기호(•)로 나열 금지
- 3개 이하 항목은 산문으로 통합
- 3단 이상 중첩 목록 금지

## 금지 문체 (영어)

- delve, leverage, utilize, harness, embark → 단순 동사
- pivotal, robust, seamless, cutting-edge, comprehensive → 구체적 묘사
- tapestry, landscape, realm(비유), synergy, testament
- "It's important to note that"
- "In today's ever-evolving world / rapidly evolving landscape"
- "In conclusion / In summary / In essence" (서두에 오는 결론)
- "Not just X, but Y" 구조 남용
- "Certainly!"로 시작하는 답변

## 구조적 AI 패턴 회피

- **획일적 문장 길이** — 모든 문장이 15~22단어면 AI. 짧은 문장(3~7)과 긴 문장(25+)을 섞어 리듬을 만든다
- **공식 단락 구조** — 모든 단락이 "주제문→설명→예시→소결론" 패턴이면 AI. 구조에 변화를 준다
- **수동태 남용** — "~이 확인되었습니다", "~가 제안됩니다" → 주체 명확 + 능동태

## HB 산출물 Voice

- **어조**: ~습니다 포멀 유지
- **문장 구성**: 긴 연결 문장 허용 (짧게 끊어 쓰려 하지 말 것)
- **혜택 서술**: benefit을 inline으로 열거 (별도 목록화보다 문장 내에 녹이기)
- **회피**: 추상 원칙 나열, 교과서적 개념 설명
- **선호**: 실제 데이터 사례 + 도메인 해석 + 옵션 table 비교

## 인간화 수정 전략 (AI 슬롭 검수 시)

AI 생성 텍스트를 사람의 목소리로 재구성할 때:

1. **목소리 구체화** — "이 기능은 효율적입니다" → "이 기능 덕분에 우리 팀은 주간 회의를 30분 줄였습니다"
2. **리듬 변화** — 짧은 문장으로 강조점 만들기. 그 다음 문장에서 맥락 채우기
3. **전환어 다이어트** — "또한" 연속 나오면 하나씩 삭제, 대부분 자연스럽게 이어짐
4. **목록 → 산문** — 3개 이하는 "A, B, C 세 가지가 핵심입니다"
5. **구체성 주입** — "많은 사람들이" → "응답자의 63%가" / "상당한 개선" → "처리 속도 2배"
6. **도입부 재작성** — AI 글 도입부는 거의 버린다. 질문·의외의 사실·역피라미드
7. **결말 재작성** — 앞 내용 반복 금지, CTA·열린 질문·직접 던지는 한 마디

## 작성·수정 후 재점검 루틴

문서를 새로 쓰거나 기존 문서에 섹션·단락을 추가한 후 반드시 다음 3항목을 재대조한다. 초기 voice treatment는 기존 텍스트에만 적용되므로, 신규 텍스트마다 재체크가 필수.

1. **Claude 습관어 grep** — 박제, 환류, 지렛대, 자가발전, 변주에 약함, 정교화, 고도화, 패러다임, 생태계(비유), 선순환, 가시화 등 LLM 특유의 관용어 검색. 1건이라도 적발되면 즉시 교체
2. **HB voice 대조** — `memory/feedback_hb_deliverable_voice.md` 재참조. ~습니다 포멀 유지, 긴 연결 문장, benefit inline 열거, 추상 원칙 나열 금지 확인
3. **섹션 간 용어 일관성** — 기존 섹션에서 이미 교체한 어휘(번역투 → 자연어 대체어)를 신규 섹션도 동일하게 적용하는지 확인
