# rules/qms-sop.md

SOP·QMS·표준 절차서·정책 문서·인증 신청서 등 구조화된 규제·관리 문서 작성 시 적용한다. 도메인은 의료기기·임상·제약·금융·교육·법무·인증 등 어느 분야든 공통.

## 단일 소스 원칙 (Single Source of Truth)

- **MD가 콘텐츠의 단일 소스**
- 수정은 MD에서 수행 → docx·pdf 재생성
- docx 직접 수정 후 MD 미반영 금지 (양방향 drift 발생)
- 변환 스크립트에 콘텐츠 데이터 하드코딩 금지 (MD 파싱만 허용)

## 문서 생성 유형

### 유형 A: 원문 문서 (새 생성)

- **대상**: SOP, 정책서, 매뉴얼, Rationale, 의견서 등 원문 문서
- **방법**: `MD 작성 → md_to_docx 변환 → docx`
- **의존성**: pandoc 3.9+, python-docx, docxcompose

### 유형 B: 양식 기입본 (채워넣기)

- **대상**: 양식에 데이터를 채우는 문서 (계획서, 체크리스트, 보고서, 평가서, 인증 신청서 등)
- **방법**: `양식 MD 복사 → 빈 칸 채우기 → 스크립트가 MD 파싱 → 양식 docx에 채움`
- **MD 작성 규칙**: 양식 MD의 **열 구조 변경 금지**. 행 추가/삭제는 허용
- **절대 금지**:
  - 양식 기입본을 pandoc으로 새로 생성 (양식 서식 손실)
  - 스크립트에 콘텐츠 데이터 하드코딩 (MD와 동기화 깨짐)
  - MD 열 구조 임의 변경 (양식 docx와 매핑 불가)

## 템플릿 보호 원칙

`templates/` 디렉토리 내 파일은 **양식 서식의 단일 소스**다. 기입본 데이터가 템플릿에 섞이면 재사용 불가.

### 직접 쓰기 금지

- **절대 금지**: `templates/` 경로 파일에 Write/Edit
- **강제 수단**: PreToolUse hook이 자동 차단 (`.claude/hooks/block-template-write.py`)
- 우회 시 즉시 HB 확인 필요 — 우회 사유 없이 hook 비활성화 금지

### 템플릿 개정 허용 조건

양식 자체를 수정하는 **양식 개정**에 한해 허용한다:

- 열 구조 변경, 항목 추가/삭제, 서식 변경 등
- **사전 조건**: 담당자 명시적 승인 후 hook 임시 비활성화 (`update-config` 스킬)
- **사후 조건**:
  1. `{stem}_changelog.md`에 개정 이유, 변경 항목, 개정일 기록 (binary-files.md changelog 규칙 준용)
  2. 문서 버전 번호 갱신 (예: Rev.1 → Rev.2)
  3. 기존 기입본 영향 여부 검토
  4. git commit 메시지에 `양식 개정` 명시
  5. 개정 완료 후 hook 즉시 재활성화

## 문서 코드 부여

- **원칙**: 문서코드는 **조직의 기존 체계 확인 후** 부여. 임의 생성 금지
- **신규 코드 부여 전 필수 확인**:
  - 해당 영역의 INDEX 또는 문서 등록부
  - 기존 문서 코드 체계 (조직마다 prefix·numbering 규칙이 다름)
  - 첨부·부속 문서 체계 (예: `{본문코드}-A{NN}` 같은 부속 코드 패턴)
- **체계 외 접두사 사용 금지**: 조직에 정의되지 않은 임의 접두사(WI-·GL-·QM- 등) 사용 금지

## 법률·규정·기준 분석

- **완독 필수**: 조항 분석 시 면제·예외·단서 규정까지 끝까지 읽을 것
- **위반 시**: 예외에 해당하는 케이스를 일반 규정으로 잘못 적용
- **범위 엄수**: 계약서·규정·요청서 검토 범위는 **명시된 문구에 한정**, 임의 확대 금지
- **레퍼런스 사양 분리**: 레퍼런스 제품·사례의 사양을 분석 대상에 그대로 투영 금지. 카테고리가 같아도 등급·위험도·적응증·맥락이 다르면 결론이 다르다

## Heading 규칙

- H1 → H2 → H3 → bullet (최대 3단계 + 글머리)
- H4 이상은 별도 Work Instruction(WI)·하위 문서로 분리

## docx 스타일

docx 생성 시 [binary-files.md](binary-files.md) 참조. 핵심:

- 본문은 heading 레벨 무관, **H1 기준 왼쪽 정렬** (indent 0)
- H1·H2·H3 색상 검정 통일
- 테이블: `#F2F2F2` 헤더 배경, bold, full grid
- Annex 섹션은 자동 넘버링 제외, 새 페이지로 시작

## 컴플라이언스·규제 산출물 검증

- SOP·정책서·법률 메모·audit 응답·regulatory 문서 작성 후 `deliverable-review` 스킬로 verification gate 통과
- 도메인별 reviewer 자동 디스패치 (예: 의료기기·임상은 `deliverable-meddev-compliance-reviewer`, 일반 spec은 `deliverable-spec-reviewer`)
- 사용자는 자기 도메인 reviewer를 `.claude/agents/`에 추가하여 디스패치 라우팅 확장 가능
