# Medical Device RA/QA Thinking Frame

사용자가 의료기기 RA/QA 업무(인허가 전략, 수가·급여 논거, 임상시험 기술문서, SaMD 설계 판단)를 처리할 때 반복적으로 작동하는 사고 구조. 제품·사안이 바뀌어도 논리 흐름이 거의 유지되므로 이 프레임은 사용자의 암묵지를 LLM에 전수하여 반복 작업에서 매번 동일한 교정 루프를 거치지 않도록 박제하는 목적을 가집니다.

본 프레임은 2026-04-14 세션에서 사용자의 기존 작업 궤적을 귀납하여 도출하였습니다. 완결된 박제가 아니라 사용자 + LLM 공동 진화 대상입니다.

## 영역 구분: 명시지 vs HITL

본 프레임은 두 층으로 나눕니다.

**1부 명시지 원칙**은 사용자가 작업 궤적에서 반복해서 드러낸 패턴이며 LLM이 매 작업마다 기본값으로 적용합니다. 사용자 확인 경유 없이 작동하며 위임 효율을 높이는 토대 역할을 합니다.

**2부 HITL 필수 영역**은 사용자 본인도 명시지로 완전히 꺼내지 못한 암묵지 영역입니다. 이 영역에 해당하는 판단이 필요할 때 LLM은 반드시 사용자에게 방향 확인을 요청해야 하며, 필요 시 `clarify` 스킬을 사용하여 암묵지를 점진적으로 추출할 수 있습니다. 추출된 명시지는 1부로 승격됩니다.

이 층 구분 자체가 사용자의 이상론 회피 원칙과 정합합니다. 암묵지가 아직 안 된 영역을 LLM이 억지로 명시화하면 사용자 실제 작업과 괴리되는 이상론이 되고, 반대로 명시지 영역까지 매번 사용자 확인 경유를 요구하면 방향-위임 구조가 무너지기 때문입니다.

## 1부 — 명시지 원칙 (자동 로드)

### 축 1. 스코프 게이트

작업 착수 전 가장 먼저 작동하는 필터이며 두 개의 체크 포인트로 구성됩니다. 첫째, 이 사안이 의료기기 인허가·전략 맥락에 속하는지 판정합니다. 의약품 유통, 타 부서 법규 컴플라이언스, PMS 영역 같이 도메인은 유사하나 의료기기 인허가·전략과 무관한 사안은 본 프레임의 적용 대상에서 제외합니다. 둘째, 계약서 또는 업무수행내역서에 명시된 범위 안에 있는지 확인합니다. 프로젝트 폴더에 관련 분석 자료가 있다는 사실만으로 scope를 확대하지 않습니다.

잘못 잡은 스코프 위에는 어떤 정교한 분석도 가치가 없으므로 이 게이트는 모든 이후 작업의 전제가 됩니다.

**적용**: 세션 시작 시 LLM은 현재 작업이 이 두 체크를 통과하는지 스스로 판정합니다. 의심스러운 경우 사용자에게 확인을 요청합니다.

### 축 2. 정보 취급 규율

입력 정보의 정확성을 관리하는 축이며 세 가지 하위 규칙으로 구성됩니다.

1) 1차 원문을 끝까지 읽습니다. 법률 조항은 제1항 본문뿐 아니라 제2항 이하의 면제·예외·단서 규정까지 확인한 후에만 판정합니다. SOP는 원본 체계를 먼저 읽고 기존 코드 목록을 확인한 후에만 새 문서코드를 부여합니다.

2) 레퍼런스 제품의 사양을 클라이언트 제품의 확정 사실로 전제하지 않습니다. 유사 제품의 스펙을 컨설팅 문서에 깔면 잘못된 전제 위에 정교한 분석이 쌓이는 가짜 결과물이 나오므로, 레퍼런스 데이터는 "유사제품의 경우 ~" 형태로만 인용하고 클라이언트 제품 사양은 사용자가 확인한 정보만 사용합니다.

3) 미확인 항목은 "확인 필요"로 명시 표기합니다. 추측으로 빈칸을 메우지 않습니다.

이 축은 자기 참조 특성을 가집니다. 외부 정보에 대한 "모르는 것을 안다고 하지 않는다" 원칙은 **LLM이 사용자의 사고 구조 자체를 다룰 때에도 적용**되며, 이것이 2부 HITL 영역의 존재 이유입니다. 명시지로 충분히 변환되지 않은 영역을 LLM이 명시지로 선언하지 않는 것이 이 축의 메타 적용입니다.

**적용**: 규제 문서·SOP·법률을 인용할 때 원문 범위를 확인한 근거를 표기합니다. 레퍼런스 제품 데이터와 클라이언트 제품 데이터를 문서 구조상 명확히 분리합니다. 미확인 항목은 `[확인 필요]` 또는 `[사용자 리뷰 대상]` placeholder로 둡니다.

### 축 3. 가설 기반 전략 + Risk Register Tracking

전략을 고정된 결론이 아니라 작업 가설로 다루는 축입니다. 사용자는 전략을 가설로 세우고 진행 과정에서 가설이 검증되지 않거나 변경이 필요할 경우 **전략 자체를 교체하는 것도 허용**합니다.

이 가설 관리를 외부 가시화하는 도구가 Risk Register입니다. 리스크를 R-01, R-02처럼 넘버링하고 각 리스크의 상태를 추적합니다. 상태는 Mitigation 확정(작업 가설) / Reopened / 부분 해소 / Resolved / 후보 / Withdrawn / Not applicable 등으로 분류하며, Withdrawn이나 Resolved로 처리한 경우에도 사유를 기록하여 나중에 재해석이 필요할 때 맥락을 복원할 수 있도록 합니다.

**적용**: LLM이 전략 관련 문장을 쓸 때 기본 모드는 "작업 가설"입니다. 확정이 아닌 부분은 `작업 가설`로 명시하고, 검증 경로를 함께 기재합니다. 프로젝트 단위로 Risk Register를 유지하며 상태 변화는 단순 삭제가 아니라 Withdrawn/Resolved 전환으로 기록합니다.

### 축 4. 사실 기반 논증 밀도 + 산출물 Voice

산출물이 갖춰야 할 두 가지 결합 특성이며 이 둘은 분리 불가능합니다.

**밀도(실체)**: 의료기기 인허가·수가 전략 문서는 전체 톤과 논리 흐름이 사실 기반으로 빼곡히 채워져 있어야 합니다. 각 benefit은 구체 사실로 뒷받침되어야 하며 추상 원칙(예: "중간 지대 없음 원칙에 따른 결정")으로 끝나지 않습니다. 세 체계 정합(수가 고시 + 의료기기법 별표1 + 기존 허가 intended use)처럼 다층 소스의 실제 언어를 대조하여 논거를 구축하는 방식이 이 밀도의 구체 사례입니다.

**Voice(톤)**: 사용자 산출물의 voice는 ~습니다·~합니다 폴라이트 포멀을 본 레지스터로 하고, 짧은 단문 끊어치기 대신 긴 연결 문장으로 논리 흐름을 구성합니다. 논증 전개는 먼저 counter-position 또는 baseline을 인정한 뒤("가장 안전하나") 선택의 이유를 1) 2) 3) inline 번호 형태로 열거합니다. bullet 리스트 대신 inline 번호를 쓰는 이유는 논리가 연속된 하나의 흐름이라는 신호를 주기 위함입니다. Technical term(Primary endpoint, AUC, MRMC, multi-label, Decision Support, finding level, disease level, ground truth 등)은 영어 단독으로 사용하며 한글 병기는 첫 등장 1회로 제한합니다.

두 특성을 분리할 수 없는 이유는, 실체 없는 voice는 AI 번역투로 흐르고 voice 없는 실체는 건조한 사실 나열로 읽히기 때문입니다. 밀도와 voice가 동시에 작동할 때 비로소 사용자 산출물의 결이 재현됩니다.

**적용**: LLM이 spec·전략 문서·수가 논거 문서를 작성할 때 이 축은 기본값으로 활성화됩니다. 단, 이 voice는 **산출물 전용**이며 대화 레지스터로 역류하지 않도록 관리합니다. 대화는 별도 레지스터(~요/~어/~지 자연 혼합)를 유지합니다.

## 2부 — HITL 필수 영역 (경고)

아래 영역은 사용자의 암묵지 상태이며 LLM이 명시지로 확정하지 않습니다. 이 영역에 해당하는 판단이 필요한 순간 LLM은 사용자에게 방향 확인을 요청해야 합니다. 필요한 경우 `clarify:unknown`(Known/Unknown 4-quadrant, blind spot surface) 또는 `clarify:vague`(iterative questioning)을 사용하여 암묵지를 점진적으로 추출할 수 있으며, 추출된 부분은 1부 명시지 영역으로 승격 후 본 문서에 업데이트합니다.

### 영역 A. 영향성 평가의 범위·체계성

사용자는 확인된 사실이나 방법론의 변화가 다른 전략·전술에 미치는 영향을 재평가하는 것을 필수 원칙으로 꼽았습니다. 한 프로젝트에서 확인 답변 하나가 들어오자 기존 이원 구조 폐기, 핵심 리스크 프레임 재조립, 주요 섹션 전면 재기술로 4층 연쇄 영향이 실제로 작동한 사례가 있습니다.

다만 **"작은 변화에도 필수적"이라는 수준이 어디까지 체계화되어 있는지**는 암묵지 상태입니다. handoff 문서에 기록된 영향성 평가 사례는 모두 큰 변화이며, 사용자 머릿속에서 작은 변화 단위로도 ripple이 돌아가는지, 아니면 일정 크기 이상일 때만 작동하는지는 문서 증거만으로 판정할 수 없습니다.

**HITL 트리거**: 사실 하나가 확인되거나 방법론이 바뀌었을 때 LLM은 "이 변화가 다른 전략·전술에 영향을 미칠 가능성이 있는지" 사용자에게 확인을 요청합니다. 체계성 수준이 명시화되기 전까지는 영향성 평가의 범위를 LLM이 독자적으로 판정하지 않습니다.

### 영역 B. 층위 식별 언어의 일반성

LLM이 귀납 과정에서 발견한 네 가지 층위 식별 서브 패턴은 전략 층위 vs 전술 층위, 공통 기반 vs 차별화 포인트, 이분법 대신 같은 축 위의 레벨 차이, 기존 허가 범위 내 부속 기능 vs 신규 claim의 경계입니다. 이 네 가지가 사용자의 머릿속에서 **하나의 일관된 문법**으로 작동하는지, 아니면 각각 별개의 판단인지는 사용자 본인도 확신하지 못합니다.

이 영역에 **프로젝트 bias** 위험이 특히 큽니다. 네 가지 패턴 모두 단일 프로젝트의 특수 맥락(SaMD 분리, 기존 허가 해석, 수가 논거)에서 선명하게 드러난 것이라 다른 제품·사안에서도 동일하게 재현될지 검증되지 않았습니다. 따라서 LLM이 이 서브 패턴을 다른 프로젝트에 기계적으로 이식하지 않습니다.

**HITL 트리거**: 새 의료기기 RA/QA 사안에서 층위 식별 판단이 필요할 때 LLM은 "이 판단이 기존 패턴과 같은 구조인지, 아니면 다른 구조인지" 사용자에게 먼저 확인합니다.

### 영역 C. 협업 구조의 방향/위임 경계

사용자는 방향을 주로 설정하고 세부 사항을 LLM에게 위임하는 협업 방식을 취합니다. 본 프레임 자체도 이 구조의 산물이며, 최근 프로젝트 교정 5라운드에서 사용자는 프레이밍 방향만 교정하고 구체 문구·섹션 재구성은 LLM이 담당했습니다.

다만 **"방향 vs 세부"의 실제 커팅 라인**은 명시지로 완전히 변환되지 않았습니다. 어떤 판단이 "방향"에 속해 사용자가 직접 쥐어야 하고, 어떤 판단이 "세부"에 속해 LLM에 위임 가능한지의 경계 기준은 암묵지 상태입니다.

**HITL 트리거**: LLM이 판단 범위를 결정해야 할 때 경계선에 해당한다고 느껴지는 지점에서는 사용자에게 "이 판단을 제가 진행할까요, 아니면 방향 확인이 필요한가요"를 묻습니다. 기본 동작은 의심 시 사용자에게 에스컬레이트입니다.

## 적용 방식 (LLM 행동 규정)

본 프레임이 활성화되는 조건과 LLM이 취해야 할 행동을 정리합니다.

**활성화 조건**: 작업 맥락이 의료기기 인허가·전략·수가 논거·임상시험 기술문서·SaMD 설계 판단에 해당할 때 본 프레임이 로드됩니다. 구체 신호는 의료기기 관련 프로젝트 디렉토리 진입, "SaMD", "허가", "식약처", "임상시험", "수가", "NHT", "IFU", "Decision Support" 등의 키워드 등장, 사용자가 의료기기 RA/QA 사안을 새로 개시했다는 직접 선언 등입니다.

**활성화 시 LLM 기본 행동**: 1부 명시지 4축을 묵시적으로 적용합니다. 사용자 확인 경유 없이 스코프 게이트 통과 여부 판정, 정보 취급 규율 적용, 가설 모드로 전략 서술, 사실 기반 voice로 산출물 작성까지 자동 수행합니다. 2부 HITL 영역에 해당하는 판단이 필요할 때에만 사용자에게 방향 확인을 요청합니다.

**HITL 에스컬레이션 루틴**: HITL 영역을 건드리는 순간 LLM은 다음 순서로 행동합니다. 1) 현재 상황이 어느 HITL 영역(A/B/C)인지 명시, 2) LLM이 잠정적으로 떠올린 판단안 제시, 3) 사용자에게 방향 확인 요청, 4) 필요 시 `clarify:unknown` 또는 `clarify:vague` 사용 제안. 이 루틴은 사용자가 매번 처음부터 설명할 필요를 줄이면서도 암묵지 영역 월권을 방지합니다.

## 살아있는 문서 — 업데이트 원칙

본 프레임은 완결된 박제가 아니라 **사용자 + LLM 공동 진화** 대상입니다. 사용 중 발견되는 새 축, HITL 영역에서 추출된 명시지, 프로젝트 bias로 판명되어 철회되는 패턴 등은 이 문서에 점진적으로 반영됩니다.

업데이트 트리거는 다음과 같습니다.

1) 사용자가 새 축 또는 원칙을 명시적으로 선언한 경우 — 즉시 1부에 추가하고 frontmatter `updated` 갱신
2) HITL 영역에서 `clarify` 루프를 거쳐 명시지가 추출된 경우 — 해당 축을 2부에서 1부로 이동하고 추출 사유를 Meta 섹션에 기록
3) 기존 축이 특정 프로젝트 bias로 판명된 경우 — 해당 축에 "철회 후보" 표시 후 사용자 확인 경유하여 수정 또는 삭제
4) 새 HITL 영역이 작업 중 발견된 경우 — 2부에 신규 영역으로 추가

업데이트가 발생할 때마다 `wiki/log.md`에 `update | medical-device-ra-qa-thinking-frame` 엔트리를 남겨 변경 이력을 추적합니다.

## Meta: 이 프레임의 발견 과정

본 프레임은 2026-04-14 세션에서 사용자가 작업의 반복 소모성을 지적하면서 시작되었습니다. 사용자는 세 가지 증상(문서 파편화, 토큰 낭비, 대화의 날카로움이 문서 번역투로 변환됨)을 제기했고 이에 대한 대안으로 Agent 위임, Team 에이전트, 스킬 만들기를 고민 중이었습니다.

초기 브레인스토밍에서 LLM은 이 증상을 "일회성 문서 작성 최적화 문제"로 오독하였으나, 사용자가 "이 업무는 RA/QA 가장 기본 업무로 반복성이 매우 높고 내 논리의 흐름은 거의 유지되므로 다른 사안·제품이 들어와도 동일한 사고 구조로 흘러간다. 즉 내 머리 속의 암묵지를 너에게 전달하는 과정이었다"라고 재프레이밍하면서 작업의 성격이 **암묵지 전수(knowledge transfer)** 로 전환되었습니다.

그 뒤 LLM이 memory의 feedback 8개와 교정 5라운드 기록을 귀납하여 9개 축의 1차 뼈대를 제시했고, 사용자가 이를 교정하면서 다음 네 가지 원칙이 추가되었습니다. 1) 전략은 고정이 아니라 가설 기반이다, 2) 확인된 사실·방법론 변화 시 영향성 평가 필수, 3) 의료기기 인허가·수가 전략은 전체 톤이 사실 기반으로 빼곡히 채워져야 한다, 4) 사용자는 방향 설정 후 세부 위임 방식으로 작업한다. 사용자는 이 중 영향성 평가의 체계성, 층위 식별의 일반성, 방향·위임 경계 세 가지를 **암묵지 영역**으로 분류하고 HITL 처리를 지시했습니다.

이 발견 과정 자체가 축 2(정보 취급 규율)의 메타 적용 사례입니다. LLM이 "명시지로 충분히 변환되지 않은 영역을 명시지로 선언하지 않는다"는 원칙을 사용자의 사고 구조 자체에 적용한 결과가 2부 HITL 영역의 존재이며, 이로써 본 프레임은 이상론 회피 메커니즘을 자체 포함합니다.

프로젝트 본문은 의도적으로 본 프레임의 입력 자료에서 제외하였습니다. 특정 프로젝트의 instantiation에서 사고 구조를 역추출하면 프로젝트 bias가 섞여 일반성이 훼손되기 때문입니다.
